# 决策对齐（防决策漂移）

> 让每次改动都与既定需求和决策对齐，而非另起炉灶。

## 1. 动手前先读
做实质改动前，先读相关的 `docs/PRD.md`（要什么）、`docs/ARCHITECTURE.md`（怎么建）、相关 `docs/adr/*`（为什么），按既定方向走，不凭空重设计。

## 2. 不静默推翻决策
- 若你的方案与某条**已接受的 ADR 相悖** → **不得在代码里静默违背**。先说明冲突、提议写一条新 ADR 取代它，经确认后再改（ADR 不可变只取代，见 `doc-sync.md`）。
- 与 PRD 需求或 ARCHITECTURE 相悖时同理：先指出，不擅自偏离。

## 3. 变更可追溯
- 每处改动应能追溯到一个需求（PRD）或缺陷。无来由的"顺手改进"不做（精准修改）。
- 提交信息说明"为什么改"，关联到需求 / 决策。

## 4. 模糊即停
- 需求不清、边界不明、存在多种解释时 → **停下来问，禁止臆测**（先思考后编码）。

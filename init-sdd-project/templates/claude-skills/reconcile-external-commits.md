# 同步外部 / 计划外提交（reconcile）

## 为什么需要它

常规迭代里每个改动走 `develop-feature` / `fix-bug` 等技能、自带 `doc-sync`，文档与代码同步。但总有提交**绕过这套流程**：逆向改造期间别人还在推、队友直接 commit、CI 自动提交、从别的分支合并进来。这些提交没人跑过 doc-sync，文档就落后、漂移。本技能是**增量版的逆向**：把这批"计划外提交"的变更补进文档，让真源重新对齐。

## 核心原则

- **只补文档、不改代码**：忠于这批提交**已经造成的现状**，不借机重构 / 改行为。
- 发现违规（破坏不变量、越界做了后续阶段的能力、与某条 ADR 相悖）→ **标出来交给用户**，不擅自改代码消除。
- 是**兜底**不是常态：常规改动该走对应技能（自带 doc-sync），别都堆给本技能事后补。

## 强制流程

### 1. 圈定提交范围
- 确定"未同步"的提交区间：跟用户确认（如 `<上次同步点>..HEAD`、某次分支合并、某几个 commit hash）。若项目用 tag / 标记记录过"上次 reconcile 到哪"，从那里到 HEAD。
- `git log --stat <range>` + 逐条看 diff，弄清每条改了什么。

### 2. 逐条分类，决定补哪份文档

| 这条提交是 | 补到 |
|---|---|
| 新功能 / 能力 | PRD 加 / 更新 FR（已实现的标 `已交付@版本`）；非平凡补 `docs/specs/` 记录 |
| 修复 | CHANGELOG |
| 重构 / 结构变化 | ARCHITECTURE（若模块 / 依赖 / 机制变了） |
| 接口变化 | API |
| 事实上的架构决策（换了库 / 改了模式） | 追认一条 ADR（状态"已接受"，背景注"现状追认"）——只追认**重大**的 |
| 用户可见变更 | CHANGELOG（未发布段；若那批提交已发版则记入对应版本段） |

### 3. 把文档补到与代码一致
- ARCHITECTURE / API **原地更新**到现状；PRD FR 表补行 / 翻状态；CHANGELOG 记账；重大决策追认 ADR。
- 忠于代码现在的样子，不是"应该的样子"。

### 4. 标出违规与漂移（关键）
- 这批提交里若**破坏 `architecture-invariants`**（引入了禁用的重型件）、**越界做了后续阶段的能力**、**与某条 ADR 相悖**——**列出来报告用户**，建议走 ADR 取代或 `rollback-change`，但**本技能不动代码**。
- 提交把代码改了但语义不清、文档无法对齐处 → 标为待澄清，问用户。

### 5. 记录"已同步到哪"
- 同步完，更新"上次 reconcile 点"标记（打个轻量 tag，或在 `.tmp/` 记最后同步的 commit），下次从这里继续，避免重复扫。

### 6. 提交
- 中文提交 `docs: 同步外部提交 <range> 到文档`，不自动 push。

## 与其他技能的关系
- 一次性把**整个现有代码库**逆向成 SDD 是 `retrofit-to-sdd`；本技能处理的是**增量**——逆向之后（或日常）少量绕过流程的提交。
- 常规改动走 `develop-feature` 等（自带 doc-sync）；本技能是兜底。

## 红线
借同步之名改 / 重构业务代码 · 文档写成"应该的样子"而非提交造成的现状 · 把破坏不变量 / 越界的外部提交**静默接受**进文档而不报告 · 给每条小提交都追认 ADR（滥用）· 自动 push。

# 升级依赖

## 核心原则

依赖升级要可控：先弄清升什么、为什么、有没有破坏性变更，再小步升、用测试兜底，绝不静默引入破坏。对接全局依赖管理准则。

## 强制流程

### 1. 范围与动机
- 明确升哪个依赖、目标版本、动机（安全公告 / 修 bug / 需要新特性）。**新增**依赖前先说明理由，不擅自引入功能重叠的包。

### 2. 读变更说明
- 看目标版本的 CHANGELOG / release notes，**识别破坏性变更**（API 改动、行为变化、最低运行时要求）。跨大版本尤其当心。

### 3. 按组件最小升级
- 精确升目标包，不无谓地一把全量升（如 `-u` / 全局 update）。
- 一次聚焦一组相关依赖，便于出问题时定位。

### 4. 验证
- 构建相关组件 + 跑全部测试全绿；留意行为变化。核心库（持久化 / 网络 / 序列化等）升级尤其要确认**没破坏架构不变量**（见 `architecture-invariants`）。

### 5. 文档与提交
- 锁文件（如 `go.sum` / `package-lock.json` / Gradle 锁等）一并提交。
- 有用户可感知影响（如最低运行时 / 平台要求变化）记 `CHANGELOG.md`。
- 中文提交 `chore(deps): 升级 X 到 vY`，不自动 push。

## 红线

不读变更说明就升大版本 · 一把全量升导致连环破坏难定位 · 升级破坏架构不变量 · 升完没跑测试。

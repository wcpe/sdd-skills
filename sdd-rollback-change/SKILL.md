# 回滚改动

## 核心原则

回滚是一次**有意的反向变更**，必须可追溯：弄清回滚什么、评估影响、用安全方式撤销、把决策与文档一并回正——不是悄悄删掉了事。

## 强制流程

### 1. 明确目标与范围
- 确认回滚对象：哪个功能 / 哪次提交 / 哪个版本，以及为什么回。范围不清先问。

### 2. 评估影响（关键，先于动手）
- 该功能是否引入了**数据模型 / 迁移、对外 API、配置项**？回滚有向后兼容代价 → **先把影响范围与迁移代价讲清**，征得确认再动（数据库 / 已发布契约的破坏性回退尤其要慎）。
- 谁依赖它？回滚会不会连带破坏别处。

### 3. 安全撤销
- 优先 **`git revert`**（新增一次反向提交），**不重写已 push 的历史**。
- 若只回滚一部分，精准撤销相关改动，保留无关进展。

### 4. 决策与文档回正
- 被回滚的功能若有支撑它的 **ADR → 写一条新 ADR 取代它**（旧 ADR 状态改"已被 ADR-NNNN 取代"，**不删**），记录"为什么回"。
- 更新 `docs/PRD.md`（需求移除 / 推迟）、`docs/ARCHITECTURE.md`、`docs/API.md`（`doc-sync`）。

### 5. 验证
- 回滚后测试全绿；被移除的行为确已消失；其余功能无损（无半截不一致状态）。

### 6. 提交
- `CHANGELOG.md` 未发布段记"移除 / 回退"；中文提交（`revert: …` 或 `feat(scope): 回退…`），不自动 push。

## 红线

重写已 push 的历史 · 有破坏性兼容影响却没先告知就回滚 · 删掉而非取代相关 ADR · 回滚后留下半截不一致状态。

# 重构

## 核心原则

重构 = **改内部结构、不改外部行为**。用测试当安全网保证行为不变，针对真实坏味道，小步前进。

## 强制流程

### 1. 安全网（先保证可验证）
- 确认目标代码有测试覆盖；覆盖薄弱 → **先补特征测试（characterization test）锁住当前行为**，再动手。
- 跑一遍现有相关测试，确认**起点全绿**——没有绿色起点就没法证明重构没破坏行为。

### 2. 明确目标坏味道
- 指出要消除的具体反模式：上帝类、长方法（>30 行）、嵌套（>3 层）、复制粘贴、魔法值、贫血模型、循环依赖等。
- 重构方向应朝 `architecture-invariants`（见该规则）：分层单向、单一职责、高内聚低耦合。

### 3. 小步重构
- 增量进行，**每一步保持测试全绿**；手法用提取方法 / 提取类、卫语句、消除重复、引入恰当模式（只在真有变化点时）。
- **严禁夹带功能变更**：重构这一次只改结构；要加 / 改行为，分别走 `sdd-develop-feature` / `sdd-fix-bug`。

### 4. 验证：行为不变
- 重构前后**同一组测试同样通过**，无行为差异。这是重构成功的唯一标准。

### 5. 文档与提交
- 仅当模块 / 依赖 / 机制结构变化时更新 `docs/ARCHITECTURE.md`；纯内部重构通常不动 PRD / API。
- 若重构实质改变了某条架构决策 → 写新 ADR 取代（`decision-alignment`）。
- 一般无用户可见变更，可不记 CHANGELOG（若有外部影响则记）；中文提交 `refactor(scope): …`，不自动 push。

## 红线

重构前没有测试安全网 · 重构里偷偷改了行为 / 加了功能 · 重构后测试结果与之前不一致 · 越改越复杂（违背简单优先）。

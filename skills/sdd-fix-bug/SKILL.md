# 修复缺陷

## 核心原则

先复现、再定位根因、最后最小修复——不靠猜、不只治标、不夹带无关改动。让"修好了"有客观证据。

## 适用边界

本技能针对**已交付 / 基线行为**的缺陷。功能还在本次开发周期、没过验证门 → 那是"还没做完"，留在 `sdd-develop-feature` 内修，别用本技能。

特殊情形——**FR 标了 `已交付`/done 但实际是断的**（单元 / e2e 绿却实机错、或从没真正工作过）：仍用本技能修，额外做两件事——① 把 PRD 状态**归真**（从没工作过的 false-done → 回退 `开发中`；真交付后回归 → 保留 `已交付@vX` 并标注损坏），并**补强验收标准**把暴露的盲区纳入；② 跨多 FR / 整条链路的非平凡修复，先按 `sdd-develop-feature` §1 写 `docs/specs/<feature>.md` 圈定范围。

## 强制流程

### 1. 复现（先写失败测试）
- **先写一个能稳定复现该 bug 的测试**（此时它失败）。不复现不动手——否则无法确认真修好，也防不住复发。
- 复现不了 → 向用户要触发条件、输入、日志、环境，不臆测。
- **复现要"复现到可回归"**：若是测试全绿却实机 / 集成才错（验证门有盲区），把失败**下沉成自动化测试**（扩 e2e / 加集成 / 模拟实机条件）；实在只能实机复现的，把"实机验收"写成 FR 的显式手动验收步骤。修完把这条测试**补进套件**，让验证门以后挡得住。

### 2. 定位根因
- 读相关代码与 `docs/ARCHITECTURE.md`，找**根因**而非表象。
- 区分这是实现 bug、对契约的误解，还是文档 / 架构本身的缺口（后者要连文档一起修）。

### 3. 最小修复
- 精准修改：只改根因处，不顺手重构、不扩范围（重构走 `sdd-refactor-code`）。
- 守 `architecture-invariants`（见该规则）：别为修一个 bug 破坏分层、违反技术约束或绕过既定边界。

### 4. 回归验证
- 第 1 步的复现测试转为**通过**；跑相关测试确认**无新回归**。
- 改到高风险区（并发、状态合并、幂等 / 守卫、失败兜底等）时复跑对应用例。
- **完成判据（声称修好前必须出示证据，否则视为未完成、继续修）**：① 复现该失败的测试 / 验收**从红转绿**；② 涉及实机 / 集成维度时，**实机复验通过**。两样拿不齐就不算修好——不轻信任何"完成了"的报告（含子代理的汇报）。实机复验被推翻（仍失败）就回到本步继续修，别因"测试已绿"提前收尾，也别切到发布 / 别的技能。

### 5. 文档与提交
- 若 bug 暴露了文档 / 契约错误，一并改（`doc-sync`）。
- §4 完成判据未全部满足不得提交（见 `git-commit.md` §3）；涉及实机维度的，待用户确认验收通过再提交。
- `CHANGELOG.md` 未发布段记修复；中文提交 `fix(scope): …`，无 AI 署名，不自动 push。

## 常见错误

| 倾向 | 纠正 |
|---|---|
| 直接改代码"看好了没" | 先写复现测试，改完用它确认 |
| 加个 `if` 兜住症状 | 找根因，否则换个输入又复发 |
| 顺手重构周边代码 | 修 bug 只修 bug，结构问题另走 `sdd-refactor-code` |
| 偶发 bug 重试几次没复现就当好了 | 偶发往往是并发 / 时序问题，要构造确定性复现 |

## 红线

没有复现测试就声称修好 · 治标不治本 · 引入回归未发现 · 为修 bug 破坏架构不变量。

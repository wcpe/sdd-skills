# 需求探讨与拆解（brainstorming）

## 核心原则

一个"完整需求"往往**模糊、跨多部分**。不盘问清楚就开干,会基于没验证的假设返工。本技能**先把需求盘清,再拆成确切的 FR、定验收、确定执行方式**——它是日常迭代的**前置门**,喂给 `sdd-develop-feature` / `sdd-parallel-develop` / `sdd-fix-bug`,但**自己不写实现代码**(同 `sdd-realign-prd` / `sdd-accept-phase` 的"只规划 / 判定不动手"分工)。

> 盘问需求的方法**参考** superpowers 的 `brainstorming` 做法(逐题澄清、多选优先、YAGNI、问清才动手),但**用我们自己的流程**——不依赖那个技能装没装,直接按下面走。

三条命脉:

1. **先盘清,后拆解** —— 目的 / 约束 / 验收 / 边界没问清,不进拆解。
2. **拆成"确切的 FR"并正确分类** —— feat 净新 / fix 修坏 / ref 重构 / 增强已交付 / docs,**别只做 feat/fix 二分**;分类错了后面路由全错(尤其别把"修已坏"塞进 PRD 当新需求)。
3. **每条 FR 带可验证验收 + 定了执行方式** —— 才算交得出手:不是"做完就行",是"怎样算做完、怎么验、由谁怎么做"。

## 适用边界

- 用于**大 / 模糊 / 多部分**的需求。**单个清楚的小功能直接 `sdd-develop-feature`、单个 bug 直接 `sdd-fix-bug`**,不必上本技能。
- 它**只盘问 + 拆解 + 定验收 + 定执行 + 登记 + 路由**,不写业务代码、不做实现。
- 它是 `sdd-parallel-develop` 的**上游**:后者要的"FR 已在 PRD + 依赖序",正由本技能产出。

## 强制流程

### 1. 盘问清需求（参考 brainstorming 的方法，走我们自己的流程）
- **先摸项目现状**:读 `docs/PRD.md`、`docs/ARCHITECTURE.md`、相关 `docs/adr/*`、近期提交——理解既定方向,再问;别脱离现状空谈。
- **逐题盘问,一次一个问题**(多选优先,开放也行):目的(解决什么)、约束、成功 / 验收标准、边界(明确**不做**什么)。不一次抛一堆、不替用户脑补。
- **先评估规模**:若这"一个需求"其实是几个独立子系统(如"做个带聊天 + 计费 + 分析的平台"),**先点破、先按子系统粗分**,别一头扎进某块的细节。
- **YAGNI**:砍掉需求里不必要的部分,不加没要求的"灵活性 / 可配置"。
- **门槛(hard-gate)**:目的 / 边界 / 验收**没问清之前,不进第 2 步**。"这需求太简单不用问"恰恰是埋假设的地方——可以问得很短,但要问。

### 2. 拆成确切的 FR + 分类（要有现状依据）
- 把盘清的需求拆成 `sdd-develop-feature` 尺寸的离散条目:每条 = 一个能**独立开发 / 测试 / 提交**的最小完整单元(够得上一个分支 / PR),落成一条**确切的 FR**(一句能力描述 + 编号占位)。
- **分类(基于现状,不凭描述拍)**:
  - **net-new 净新能力**(代码里还没有) → **feat** → 新 FR
  - **修已坏**(存在但不工作 / 报错 / 不符) → **fix** → bug(**通常不进 PRD §4**,进 CHANGELOG);例外:"某 FR 标了 `已交付` 但其实坏了" → **归真**(FR 退回 `开发中`)
  - **增强已交付 FR** → 视情况:加新 FR,或在原 FR 上扩(动了架构决策要配 ADR)
  - **纯结构 / 可读性**(行为不变) → **ref / refactor**
  - **纯文档** → **docs**

### 3. 为每条 FR 定明确验收标准（"验收 FR"）
- 每条 feat FR 写**可验证的验收标准**(将来进 PRD §6 / 该 FR 的 `docs/specs/<feature>.md`),让 `sdd-develop-feature` 测试先行有靶子、`sdd-accept-phase` 有据可依。
- 涉及实机 / 集成 / 真实环境维度的,验收标准里**点明"要真机过"**(`测试绿 ≠ 真能用`)。
- 没有可验证验收的 FR 不算拆完——"做完就行"是红线。

### 4. 排依赖序（为执行方式做准备）
- 标出条目间**依赖**(A 要等 B 先落)——它决定能不能并行,是下一步定执行方式与 `sdd-parallel-develop` 的输入。

### 5. present + 用户确认（登记前的闸）
- 把结果列给用户:**FR 清单**(分类 + 验收标准 + 依赖序 + 执行方式建议)。
- 有设计岔路时给 **2-3 个方案 + 推荐**(同 brainstorming),让用户拍。
- **用户确认后才登记**;不擅自改 PRD、不擅自起开发。

### 6. 确定执行方式 + 登记 + 路由（确认后）
- **确定执行方式**:
  - 1 条 feat → `sdd-develop-feature` 单跑
  - ≥2 条**无相互依赖**的 feat → `sdd-parallel-develop`(依赖序已备好;它会再问 rebase / merge 等)
  - 有依赖链 → 分批:可并行的先一批,被依赖的等前批落 main 再开
  - fix → `sdd-fix-bug`;ref → `sdd-refactor-code`;docs → `sdd-update-docs`;归真 → `sdd-fix-bug`
- **登记**:feat FR 加进 `docs/PRD.md` §4(状态 `计划`、贴期、附验收草拟);fix / ref / docs 记进计划的待办段(fix 不进 PRD §4)。
- **产出**:计划写 `.tmp/brainstorm-<需求名>-<日期>.md`(**不入库**):FR · 分类 · 验收 · 依赖序 · 执行方式 · 路由。

## 与其他技能的关系
- **盘问方法**参考 superpowers 的 `brainstorming`,但**用我们自己的流程、不依赖它**(它没装也照跑)。
- **下游**:产出的 FR + 依赖序喂 `sdd-develop-feature`(单个)/ `sdd-parallel-develop`(批量);fix 喂 `sdd-fix-bug`。
- **与 `sdd-realign-prd` 对称**:本技能**吃新需求、盘清拆解登记**;`sdd-realign-prd` 整理**已存在**的 PRD 结构。
- 它是**日常迭代循环的前置门**——之前循环假定你已知"这是一个 feat / 一个 fix",本技能补上"先把大需求盘清、拆成带验收的确切 FR"。

## 红线
需求没盘清就开拆(基于臆测)· 只做 feat / fix 二分而漏掉 ref / 增强 / 归真 / docs · 把"修已坏"误当净新能力塞进 PRD 当新 FR · 拆出的 FR 没有可验证验收标准("做完就行")· 拆完不经用户确认就改 PRD / 起开发 · 借拆解之名直接写实现代码 · 把计划入库(应留 `.tmp/`)· 给一个清楚的小需求硬套本技能(直接 `sdd-develop-feature` / `sdd-fix-bug` 即可)。

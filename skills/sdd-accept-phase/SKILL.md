# 验收 / 发布就绪审计（整期 · 版本 · 改动）

## 核心原则

"做完没做完"，看的是范围内**每条 FR 是否都满足其验收标准、且有可验证证据**——不是"感觉做完了"。这是单功能"完成判据"的**成批版**：整期、一版待发、一批改动，都按同一把尺。本技能**只判定 + 报告 + 路由**，不顺手改业务代码、不改需求。

## 适用边界

- **三种粒度**（见 §1 的 scope）：**整期收口** / **当前版本待发** / **一批改动自审**。单个功能的完成判定在 `sdd-develop-feature` / `sdd-fix-bug` 内部走，不必每个都开本技能；但**攒了 ≥2 个 FR、一个 PR、或一版待发**时用它整批扫一遍。
- **基准是 PRD**（§4 FR 表 + §6 验收标准 / 各 FR 的 spec），**不是 `.tmp/实施计划`**（那只是任务清单、会过期、不入库，仅作辅助核对）。
- **真机标准不随范围缩小而放低**：哪怕只验一个改了 FR-7 审计的 PR，涉及"审计中可查"就要真发一次、看 /audits 真有那条——`测试绿 ≠ 真能用` 是这把尺的灵魂，**缩范围不缩标准**。

## 强制流程

### 1. 圈定验收范围（先定 scope，三选一）

开工先和用户确认验收哪一档；不清就问。三种 scope **共用后面 §2~§5 的判定 / 报告 / 路由逻辑**，只是"圈哪些 FR / 改动"的方式不同：

- **`scope: phase`（整期，默认）** —— PRD §4 表里**某一期**（如 P1）的全部 FR。用于整期收口、发版前过一遍。
- **`scope: version`（当前版本待交付）** —— 本次准备发的版本涉及的 FR：取 PRD §4 状态为 `开发中` 的 FR **∪** `CHANGELOG.md`「未发布段」记到的能力。用于"这版要发的东西够不够格"，**不必等整期**。
- **`scope: changes`（当前改动）** —— 一段 git 改动触及的 FR / 验收点：
  - 默认范围 = 上个版本 tag 到 HEAD（`git describe --tags --abbrev=0`..`HEAD`）；或用户指定 base（某分支 / commit / `--staged` / 某 worktree 分支）。
  - `git diff --stat <base>..HEAD` 看改了哪些文件，**反查触及的 FR**（改动落在哪个模块 / spec / PRD 行）。
  - 用于 PR 合并前自审、worktree 落地前自检、攒几个 commit 想发修订版前的把关。

无论哪种 scope，都列出范围内每条 FR（或改动点）及当前状态（`计划` / `开发中` / `已交付`）。

### 2. 逐条核对（要证据）
对范围内**每个 FR**，判 **通过 / 不通过 / 部分**，并给证据：
- **对照验收标准**：走 PRD §6（或该 FR 的 `docs/specs/<feature>.md`）里对应判据。
- **测试证据**：覆盖该 FR 的单元 / 集成 / E2E 测试存在且**全绿**（判据见 `testing-and-quality`）。
- **真机证据**：涉及实机 / 集成 / 真实环境的（如"杀进程仍可用""热更秒级生效"），**真机跑过**——`测试绿 ≠ 真能用`，别让自动化掩盖真实失败。
- 区分"代码没实现""实现了但不符验收""只是没测 / 没验"——它们后续路由不同。

### 3. 产出验收报告
- 写一份逐项的**通过 / 差距清单**到 `.tmp/`（**不入库**），文件名按 scope 区分、便于对照历史：
  - phase → `.tmp/acceptance-phase-<期>-<日期>.md`
  - version → `.tmp/acceptance-version-<目标版本>-<日期>.md`
  - changes → `.tmp/acceptance-changes-<base>..HEAD-<日期>.md`
- 每条 = 判定 + 证据 + 差距描述。
- 给一句结论：**范围内是否全部通过** = 是否达到该 scope 的就绪（整期可收口 / 版本可发 / 改动可合）。

### 4. 路由不通过项（本技能不改代码）
- **没实现 / 与 PRD 不符** → `sdd-fix-bug`（先写复现不符的测试）或回 `sdd-develop-feature` 续做；该 FR 保持 `开发中`。
- **PRD 写错了**（代码对、文档过时）→ `sdd-update-docs` 改 PRD。
- **要砍 / 推迟** → 跟用户确认；必要时 `sdd-rollback-change`，或把该 FR 的"期"挪到后续。

### 5. 收口或阻断（按 scope）
- **全部通过：**
  - `phase` / `version` → 交给 `sdd-release-version` 发版（把交付的 FR 标 `已交付@vX.Y.Z`、CHANGELOG 分段、打 tag）。
  - `changes` → 该批改动**可放行**：合 PR / 落 worktree 到 main / 继续攒版（视用户意图）。本技能**不自动合、不自动 push**，只给"可放行"结论。
- **未全部通过 → 列出阻塞项、不放行**（不发版 / 不合并）；阻塞项修到过验收后，回第 2 步复核。

## 与其他技能的关系
- 它是 `sdd-release-version` 的**前置门**（`phase` / `version` scope）：本技能管"够不够格发"（逐条 PRD 验收），release-version 管"怎么发"（SemVer / CHANGELOG / tag）。
- `changes` scope 是 `sdd-parallel-develop` 落地前、PR 合并前的**自审闸**。
- 单功能完成判定 → `sdd-develop-feature` / `sdd-fix-bug`；本技能是它们的**成批汇总闸**。

## 红线
没有逐条证据就宣布"完成 / 可发 / 可合" · 拿"测试绿"冒充"验收过"（漏真机 / 集成维度）· **范围缩小就放低真机标准** · 把未通过的 FR 标 `已交付` · 借验收之名改业务代码或改 PRD 需求 · 有阻塞项却放行（发版 / 合并）。

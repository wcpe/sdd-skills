# 期验收 / 发布就绪审计

## 核心原则

一整期"做完没做完"，看的是 PRD 里这期**每条 FR 是否都满足其验收标准、且有可验证证据**——不是"感觉做完了"。这是单功能"完成判据"的**整期版**。本技能**只判定 + 报告 + 路由**，不顺手改业务代码、不改需求。

## 适用边界

- 用于**整期 / 里程碑 / 发版前**的成批验收。单个功能的完成判定在 `sdd-develop-feature` / `sdd-fix-bug` 内部走，不必每个都开本技能。
- **基准是 PRD**（§4 FR 表 + §6 验收标准 / 各 FR 的 spec），**不是 `.tmp/实施计划`**（那只是任务清单、会过期、不入库，仅作辅助核对）。

## 强制流程

### 1. 圈定验收范围
- 默认 = PRD §4 FR 表里**某一期**（如 P1）的全部 FR，或某版本计划交付的 FR 集。范围不清先问。
- 列出范围内每条 FR 及当前状态（`计划` / `开发中` / `已交付`）。

### 2. 逐条核对（要证据）
对范围内**每个 FR**，判 **通过 / 不通过 / 部分**，并给证据：
- **对照验收标准**：走 PRD §6（或该 FR 的 `docs/specs/<feature>.md`）里对应判据。
- **测试证据**：覆盖该 FR 的单元 / 集成 / E2E 测试存在且**全绿**（判据见 `testing-and-quality`）。
- **真机证据**：涉及实机 / 集成 / 真实环境的（如"杀进程仍可用""热更秒级生效"），**真机跑过**——`测试绿 ≠ 真能用`，别让自动化掩盖真实失败。
- 区分"代码没实现""实现了但不符验收""只是没测 / 没验"——它们后续路由不同。

### 3. 产出验收报告
- 写一份逐 FR 的**通过 / 差距清单**到 `.tmp/`（**不入库**）：每条 = 判定 + 证据 + 差距描述。
- 给一句结论：**本期是否达到发布就绪**（范围内 FR 全部通过才算）。

### 4. 路由不通过项（本技能不改代码）
- **没实现 / 与 PRD 不符** → `sdd-fix-bug`（先写复现不符的测试）或回 `sdd-develop-feature` 续做；该 FR 保持 `开发中`。
- **PRD 写错了**（代码对、文档过时）→ `sdd-update-docs` 改 PRD。
- **要砍 / 推迟** → 跟用户确认；必要时 `sdd-rollback-change`，或把该 FR 的"期"挪到后续。

### 5. 收口或阻断
- **全部通过 → 交给 `sdd-release-version`** 发版：把本期交付的 FR 标 `已交付@vX.Y.Z`、CHANGELOG 分段、打 tag。
- **未全部通过 → 列出阻塞项、不放行**；阻塞 FR 修到过验收后，回第 2 步复核。

## 与其他技能的关系
- 它是 `sdd-release-version` 的**前置门**：本技能管"够不够格发"（逐条 PRD 验收），release-version 管"怎么发"（SemVer / CHANGELOG / tag）。
- 单功能完成判定 → `sdd-develop-feature` / `sdd-fix-bug`；本技能是它们的**整期汇总闸**。

## 红线
没有逐条证据就宣布"整期完成" · 拿"测试绿"冒充"验收过"（漏真机 / 集成维度）· 把未通过的 FR 标 `已交付` · 借验收之名改业务代码或改 PRD 需求 · 有阻塞项却放行发版。

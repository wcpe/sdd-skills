# PRD 规范对齐(realign)

## 核心原则

把 PRD 修回 SDD 规范(期 / 版本 / FR 状态 / ADR 演进规则,见 README「期 · 版本 · FR 状态 · ADR 怎么演进」)是**结构性整改**,不是常规迭代。本技能负责**诊断与开工单**——按问题分级、给现状/目标对照、路由到具体执行技能;**它本身不批量改 PRD、不动业务代码**,避免"想顺便理一下"反而把另一处搞乱。

## 适用边界

- 用于**已经有 PRD 但结构 / 状态不合规**的情况;不替代:
  - `retrofit-to-sdd`(给完全无治理的项目套上 SDD)
  - `sdd-accept-phase`(基于规范 PRD 做整期验收,要测试+真机证据)
  - `sdd-update-docs`(逐项纯文档改写)
- 它是它们的**前置闸**:**先把 PRD 修对,再谈验收和迭代**——否则 sdd-accept-phase 拿不可信的 PRD 当基准会得出"看似全过"的假结论。

## 强制流程

### 1. 八类诊断清单(逐项扫描 `docs/PRD.md`)

对照八类常见结构问题,逐条标 ✅ / ❗ / 🟡:

| # | 检查项 | 阻断 / 严重 / 提示 |
|---|---|---|
| 1 | **期号 / 优先级 / 版本 三套体系是否叠加**(同时存在 P0/P1 大节 + 每个 FR 的"优先级"字段 + V1/V2 版本分组) | ❗ 阻断 |
| 2 | **期号命名是否一致**(P0 与 P1 混用、零填充 phase01、中途换风格、期号被当版本用) | ❗ 阻断 |
| 3 | **FR 编号唯一性**(重复编号、跳号如 FR-100~106、编号空间被分组锁死) | ❗ 阻断 |
| 4 | **状态可信度**(✅ done 但验收清单大半 `[ ]`、误标 done 未归真、状态枚举里有 deprecated / deferred 但从没用过) | ❗ 阻断 |
| 5 | **§7 分期主题是否存在**(只有期号标签、没说每期"主题 / 目标") | ❗ 阻断 |
| 6 | **§6 整期验收是否成形**(验收只散在每个 FR 下,没有期级整批清单 → sdd-accept-phase 扫不到) | 🟡 严重 |
| 7 | **PRD 错位内容**(bug 记录、实施任务、过程性笔记、临时备注进了 PRD) | 🟡 严重 |
| 8 | **死字段 / 过期链接**(字段长期没人填、链接 404、关联 ADR 已被取代未更新) | 🟢 提示 |

### 2. 产出诊断报告 + 工单

写一份 `.tmp/prd-realign-<日期>.md`(**不入库**,按 SDD 第 9 条规矩),包含:

- **结论速览**:符合 / 部分符合 / 不符合;阻断级问题数。
- **逐项诊断**:每条问题给 ① 现状(引原文片段)② 应改成什么(目标态)③ 走哪个技能执行 ④ 必要时给修改前后对照。
- **工单清单**:按"❗ 阻断 → 🟡 严重 → 🟢 提示"排序,每项 checkbox 可勾。

### 3. 路由(本技能只产出工单,执行交给其他技能)

| 问题类 | 走哪个技能执行 |
|---|---|
| FR 编号重复 / 跳号 / 编号空间整理 | `sdd-update-docs` |
| 期 / 优先级 / 版本错位的清理与选型 | `sdd-update-docs` |
| §7 分期主题补全、§6 整期验收清单成形 | `sdd-update-docs` |
| 误标 done 的 FR **状态归真**(从没工作过→开发中;真交付过后回归→保留版本号+标"回归损坏") | `sdd-fix-bug`(其适用边界明确写了归真规则) |
| PRD 里错位的 bug 记录 → 移出 | `sdd-update-docs`(移出)+ `sdd-fix-bug`(本身真未修则修) |
| 死字段 / 过期链接 / ADR 已被取代未更新 | `sdd-update-docs` |
| 修完后是否够做整期验收 / 发版 | `sdd-accept-phase` |

### 4. 顺序提醒(关键)

- **先归真状态,再整理结构**——状态决定哪些 FR 现状是 done / in-progress / todo,直接影响 §7 分期归属与 §6 验收策略。先把假 done 翻回去,再排期、补 §6,顺序反了会把"待修"FR 当成"已交付"分到旧期。
- **PRD 改完不立即改业务代码**——本技能产出工单后,用户在另一个会话按工单走 `sdd-update-docs` / `sdd-fix-bug` 等技能逐项执行,职责分清。
- **不新增期号扛改动**——见 README「期号命名」节,不允许 P4 = "拨乱反正"、P5 = "重做" 这种用法。

### 5. 提交与持续追踪

- 工单本身放 `.tmp/`,**不入库**;每完成一项划掉一项。
- 工单产生的实际改动按对应技能的 `git-commit` 规范分多次提交(`docs(prd): ...` / `fix(...): FR-X 状态归真` 等),不要堆成一个巨型 commit。
- 全部工单清零后,跑 `sdd-accept-phase` 出一份基于**已规范化 PRD** 的整期验收报告,验证修对了没。

## 与其他技能的关系

- **`retrofit-to-sdd`**:针对无治理项目;本技能针对已有 PRD 但结构错的项目。
- **`sdd-update-docs` / `sdd-fix-bug`**:本技能产出的工单的执行者。
- **`sdd-accept-phase`**:本技能的下游收口。先 realign 让 PRD 可信,再 accept-phase 验收。
- **`sdd-reconcile-external-commits`**:针对"代码绕过流程进了,文档没跟上";本技能针对"文档自身规范错位",两者方向相反。

## 红线

借诊断之名擅自批量改 PRD / 业务代码 · 把 bug 记录留在 PRD 不移除 · 没归真状态就重整分期(会把待修 FR 当成已交付分到旧期) · 用新增期号(P4/P5...)扛改动 · 工单产出后不执行就声称"PRD 已规范化" · 跳过 `sdd-accept-phase` 直接发版。

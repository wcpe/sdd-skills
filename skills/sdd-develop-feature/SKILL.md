# 开发新功能

## 核心原则

新功能要落在既定方向上，而不是另起炉灶：先对齐需求与架构、确认它在当前阶段范围内、用可验证的目标驱动实现，最后让文档与代码保持一致。完整流程见 `docs/CONTRIBUTING.md`，红线见 `.claude/rules/`。

## 强制流程

### 1. 对齐与范围（动手前）
- 读 `docs/PRD.md`、`docs/ARCHITECTURE.md` 及相关 `docs/adr/*`，理解既定方向（见 `decision-alignment`）。
- 判断该功能属于哪个阶段：若它是后续阶段才计划的能力，**停下来和用户确认是否现在做**（见 `scope-discipline`），不擅自越界、不为它埋占位字段。
- 需求或边界不清 → 先问，不臆测。
- **非平凡功能先写规格**：够得上一个分支 / PR 的功能，先复制 `docs/specs/_template.md` 到 `docs/specs/<feature>.md` 写清需求 / 设计 / 任务 / 验收，再往下走；小改动免（见 `docs/specs/README.md`）。

### 2. 更新 PRD（要什么）
- 在 `docs/PRD.md` 增 / 改对应功能需求（FR）与**验收标准**，标好优先级；新需求加一行、状态置 `计划` 或 `开发中`（交付时由 `sdd-release-version` 标 `已交付@版本`）。这是判断"做完没做完"的依据。

### 3. 架构与契约（怎么建）
- 涉及架构决策（引入新技术 / 新模式 / 推翻旧决策）→ 写一条新 `docs/adr/NNNN-*.md`（必要时取代旧 ADR，旧的标记"已被取代"，不删）。
- 更新 `docs/ARCHITECTURE.md`（新模块 / 数据模型 / 机制 / 依赖）；接口变更同步 `docs/API.md`。

### 4. 目标驱动：测试先行
- 把"加功能"转成可验证目标：**先为新行为写测试**（此时应失败），再写实现让它通过。这样"做完"有客观标准。
- 高风险区（并发、状态合并、幂等 / 守卫、失败兜底等）尤其要覆盖（见 `testing-and-quality`）。

### 5. 实现
- 最少、最直接的代码解决问题，不实现需求以外的东西。
- 守 `architecture-invariants`（见该规则）。
- 精准修改：只动该动的，不顺手改无关代码。

### 6. 验证
- 新测试通过 + 相关测试全绿；满足验证门（判据见 `testing-and-quality.md` §1：PRD 验收 + 高风险区测试 + 组件测试绿）。
- **门没过 = 没 done**：验证门没满足就是"功能还没做完"，在本周期内继续修，**别因此另开 `sdd-fix-bug`**（那是给已交付 / 基线行为的缺陷用的）；只有功能交付后才暴露的缺陷才走 `sdd-fix-bug`。
- **门要反映"真实可用"**：别让"测试绿"替代"真的能用"。已知有实机 / 集成 / 真实环境维度的，把它纳入 FR 验收与测试——否则单元 / e2e 全绿也不算 done。
- **声称 done 前出示证据、不轻信报告**：宣布功能完成必须摆出验证门各项的实际运行结果（命令 + 通过情况、实机 / 集成维度的复验），拿不齐就是没做完、继续修；不凭"跑通了"的口头结论（含子代理的汇报）。证据与提交门控见 `testing-and-quality.md` §1。

### 7. 文档同步与提交
- 按 `doc-sync` 自检：PRD / ARCHITECTURE / API 是否还与代码一致。
- `CHANGELOG.md` 未发布段记本次新增，引用具体模块 / 能力。
- 按 `git-commit` 规范写中文提交（`feat(scope): …`，无 AI 署名）；**验证门未全绿不得提交**（见 `git-commit.md` §3）；不自动 push，提交由用户确认。

## 红线

后续阶段能力未确认就实现 · 与某条已接受 ADR 冲突却静默违背 · 跳过测试 · 改完不同步文档。出现即停止并确认。
